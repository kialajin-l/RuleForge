关于MVP版本与最终版的差异

## 1️⃣ 关于"收集用户反馈"

**目前的能力：**

✅ **可以做到**：
- 用户手动使用 `ruleforge extract` 提取规则
- 用户通过插件命令触发规则提取
- 规则保存到本地 `.ruleforge/rules/` 目录
- 用户手动提交至 GitHub（通过 CLI 或手动 PR）

❌ **暂未实现**：
- **自动收集反馈**：插件还没有自动监听会话结束并弹窗提示
- **自动化触发**：需要用户手动执行命令
- **远程反馈收集**：没有服务器接收用户行为数据

**原因**：
Day 7 完成的插件是**基础版**，只有：
- 命令注册（`ruleforge.extract`）
- 文件保存事件监听（记录日志，但还没集成 RuleForge 核心引擎）
- 配置读取

**缺少**：
- Webview 确认面板（UI 还没做）
- 与 @ruleforge/core 的桥接层（bridge 目录是空的）
- 实际的规则提取调用

## 2️⃣ 关于服务器 vs GitHub

你的记忆是对的！我们确实讨论过服务器方案：

### **之前的规划**（PRD 中的混合架构）

```
方案A：纯GitHub（MVP阶段）✅
- 规则内容：GitHub 仓库
- 分发：npm + VSIX
- 贡献：用户手动 PR
- 成本：$0

方案B：混合架构（V1.0）⏸️
- GitHub：规则内容存储
- 轻量云服务：
  * API Gateway（Cloudflare Workers）
  * 事件处理器（Cloud Functions）
  * 向量索引（Pinecone）
  * 元数据DB（Supabase）
- 成本：$20-100/月
```

### **现在的状态**

| 功能 | MVP方案 | 未来方案 |
|------|---------|---------|
| **规则存储** | GitHub 仓库 | GitHub 仓库（不变） |
| **规则分发** | npm / VSIX | 增加 CDN 加速 |
| **用户贡献** | 手动 PR | 自动化 PR 创建 |
| **反馈收集** | GitHub Issues | 云端事件追踪 |
| **规则搜索** | 本地关键词 | 云端语义搜索 |
| **服务器** | ❌ 不需要 | ✅ 轻量云服务 |

## 🎯 总结

**现在（MVP）：**
- ✅ 完全基于 GitHub，无需服务器
- ✅ 用户手动操作（CLI/插件命令）
- ✅ 本地规则提取和存储
- ⚠️ 需要用户主动使用，不会自动收集

**未来（V1.0）：**
- 🔄 引入轻量云服务（$20-100/月）
- 🔄 自动监听 + 弹窗提示
- 🔄 云端规则搜索和推荐
- 🔄 自动化贡献流程
